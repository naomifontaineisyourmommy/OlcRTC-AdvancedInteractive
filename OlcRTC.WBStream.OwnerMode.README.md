# olcrtc — WB Stream owner-mode (для панели управления)

> Описывает внешнее взаимодействие с локальным патчем **WB Stream owner-mode**:
> `srv` сам создаёт комнату WB Stream и держит её открытой, печатая её ID в
> stdout. Внутренняя реализация и код неважны — важно, **как запускать процесс,
> что подавать на вход (конфиг) и что читать из его вывода**.

---

## 1. WB Stream owner-mode: автосоздание и удержание комнаты

### Что это даёт снаружи
Раньше для WB Stream нужно было вручную создать комнату в вебе и вписать её ID в
конфиг. Теперь `srv` **сам создаёт комнату** по токену аккаунта и **держит её
открытой** всё время своей работы, а её ID **печатает в stdout** — панель его
подхватывает и раздаёт клиентам.

### Что подать на вход (поля YAML-конфига)
```yaml
mode: srv
auth:
  provider: wbstream
  token: "<WB ACCESS TOKEN>"   # опциональное поле: bearer-токен аккаунта WB
net:
  transport: vp8channel
  dns: 8.8.8.8:53
vp8:
  fps: 60
  batch_size: 64
crypto:
  key: "<64 hex>"
data: data
# room.id НЕ указывается — srv создаст комнату сам
```

- **`auth.token`** — опциональное поле. Это bearer-токен аккаунта WB Stream
  (берётся из залогиненной сессии аккаунта). Задаётся **строкой прямо в YAML**,
  отдельного файла токена нет.
- **Транспорт для WB — только видео** (`vp8channel`; также `videochannel`/
  `seichannel`). `datachannel` поверх WB Stream не работает — туннель не
  поднимется.
- Поведение в зависимости от `auth.token` и `room.id`:

  | `auth.token` | `room.id` | что делает srv |
  |---|---|---|
  | задан | пуст | **создаёт новую комнату**, владеет ею, печатает её ID в stdout |
  | задан | указан | подключается к указанной комнате как владелец (не создаёт) |
  | пуст | указан | обычный режим (заходит как гость) |
  | пуст | пуст | ошибка запуска (нужен `room.id`) |

### Что прочитать из вывода
Сразу после старта `srv` (когда токен задан и `room.id` пуст) в **stdout**
появляется ровно одна строка:

```
Created and connected to WB Stream room id: <ROOM_ID>
```

`<ROOM_ID>` — это идентификатор созданной комнаты (UUID-подобный, напр.
`019e9dcd-c7e5-727c-8a56-54a1356bd511`).

### Операционные правила (обязательно учесть в панели)
- **Комната живёт, пока жив процесс `srv`.** Владелец = этот процесс. Если `srv`
  упал/остановлен — комната закрывается, клиенты отваливаются.
- **При каждом перезапуске процесса создаётся НОВАЯ комната** (новый `<ROOM_ID>`).
  Панель должна заново считать ID из stdout после рестарта и заново раздать
  клиентам. Старый ID становится невалидным.
- **Токен**: один аккаунт = один `srv`. Токен долгоживущий, но может быть отозван
  сервером WB; тогда создание комнаты/реконнект перестанут работать — панели
  стоит детектить это (см. ниже) и сигналить «нужен новый токен».

### Как панель видит проблему с токеном/реконнектом
Если токен невалиден или сеть выхода завёрнута неправильно, в **stderr** будут
повторяющиеся строки вида:
```
... livekit reconnect failed: ... get token: ...
```
и комната не поднимется. Это сигнал «проверь токен/сеть этого инстанса».

## 2. Пример вывода (что увидит панель)

**stdout:**
```
Created and connected to WB Stream room id: 019e9dcd-c7e5-727c-8a56-54a1356bd511
```

**stderr (фрагмент, с префиксом времени):**
```
2026/06/06 23:39:31 Link connected
```

> `Link connected` в stderr означает, что srv реально поднял owner-соединение и
> комната открыта. Дальше идёт обычный лог. TURN-строки вида
> `failed to allocate on TURN ... 401 Unauthorized` — безвредный шум (соединение
> встаёт по рабочему ICE-кандидату), на работу не влияют.

## 3. Шпаргалка по парсингу (стабильные подстроки)

Панели стоит матчить по этим подстрокам (форматы лога в принципе могут меняться —
ориентируйся на якорные подстроки, а не на точную позицию):

| Что | Поток | Якорь / шаблон |
|---|---|---|
| ID созданной комнаты | stdout | `Created and connected to WB Stream room id: (<id>)` |
| Линк сервера поднят | stderr | `Link connected` |
| Проблема токена/реконнекта (WB) | stderr | `livekit reconnect failed` |

## 4. Краткий чек-лист для панели

- Запуск инстанса WB-owner: сгенерировать YAML с `mode: srv`,
  `auth.provider: wbstream`, `auth.token: <токен>`, `net.transport: vp8channel`,
  общий `crypto.key`, без `room.id`.
- После старта прочитать из **stdout** строку `Created and connected to WB Stream
  room id: <id>` → это ID комнаты → раздать клиентам (`cnc`) в их `room.id`
  (+ тот же `crypto.key`, тот же `transport`/`vp8`).
- Держать процесс `srv` живым всё время сессии (он — владелец комнаты).
- На рестарт — заново считать новый ID из stdout и перераздать клиентам.
- Деградация токена — повторяющийся `livekit reconnect failed` в stderr →
  пометить инстанс как требующий нового токена.
